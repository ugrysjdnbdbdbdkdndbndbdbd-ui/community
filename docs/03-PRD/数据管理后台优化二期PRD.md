## 二、优化二期目标

- **可用性**：在无后端、file:// 打开等场景下，核心流程仍可完成，且提示清晰。  
- **体验**：减少无效报错、默认值误导，操作有反馈、可预期。  
- **可维护性**：采集与完整度逻辑清晰，便于后续接入真实 API 与扩展字段。

## 三、优化二期功能需求

### 3.1 体验与反馈（P0）

| 编号 | 需求 | 描述 | 验收标准 |
|------|------|------|----------|
| O2-1 | 采集前 API 可用性检测 | 用户/商业数据「开始采集」前，先请求一次 /api/health（或轻量采集接口），若不可用则直接走本地回退并单条提示，避免逐条超时 | 未启动后端时，仅一条「后端未启动，已使用本地数据补充」类提示 |
| O2-2 | 案例列表加载态 | 列表首次加载与刷新时显示 loading（骨架或文案），避免长时间白屏或旧数据残留 | 有明确加载中状态，加载完成后消失 |
| O2-3 | 保存/同步成功反馈 | 单条保存、批量保存、数据同步、采集完成等操作成功后，有 Toast/Inline 成功提示，并可选自动刷新列表 | 用户能明确知道「已保存/已同步」 |
| O2-4 | 错误信息可读 | 接口/DB 报错时，前端展示简短、可操作的文案（如「网络异常，请检查后端服务」），并可在控制台或「采集日志」中查看详情 | 不直接暴露堆栈，但有途径查因 |

### 3.2 数据与展示（P0）

| 编号 | 需求 | 描述 | 验收标准 |
|------|------|------|----------|
| O2-5 | 完整度计算统一 | 全站（列表、详情、质量/分析页）使用同一套完整度规则：5D 40%、用户数据 30%、商业 15%、关键指标 15%，且 DB 与 JS 合并时取 max | 与《数据完整度默认展示40%》优化一致，无矛盾口径 |
| O2-6 | 空状态与零数据 | 无案例时展示「暂无案例，请先同步硬编码数据」等引导；某次采集 0 条时，日志中说明原因（如「全部为本地回退且无对应字段」） | 无歧义的空状态与 0 条结果说明 |
| O2-7 | 分页与性能 | 案例列表分页正常；单页条数可选（如 20/50/100），总条数、当前范围展示正确 | 大量案例下列表不卡顿，分页信息正确 |

## 四、非功能需求

- **兼容性**：保持与现有 community_cases.js、社区推导看板、localStorage 联动的兼容；不破坏 file:// 与 npx serve 双模式。
- **性能**：列表渲染与分页在 1000 条级案例下可接受；SQL.js 与 localStorage 回退切换不卡死。
- **安全**：不在前端写死密钥；API 基地址可配、不强制写死端口。

---

## 五、技术要点（建议）

- **API 基地址**：`window.CASES_API_BASE = window.CASES_API_BASE || location.origin`，采集请求使用 `CASES_API_BASE + '/api/collect/...'`，可在 HTML 或脚本开头覆盖。
- **完整度**：继续使用当前 40+30+15+15 权重；任何合并处均用 `Math.max(dbCompleteness, jsCompleteness)`，避免 DB 覆盖 JS 更高完整度。
- **采集流程**：先 `fetch(apiBase + '/api/health')`，失败则不再逐条请求后端，直接走本地回退并打一条汇总日志。

## 七、成功指标

- **可用性**：在仅 file:// 或仅 npx serve（无 Node 后端）情况下，用户能完成「查看列表、同步硬编码、采集用户/商业数据（本地回退）、导出」全流程，且无成片报错。
- **体验**：数据完整度与真实数据一致，无「全列 40%」等默认值误导；关键操作有反馈、有确认。
- **可维护性**：采集与完整度逻辑集中、可配置，便于后续接入真实采集 API 与扩展《数据管理后台二期PRD》中的质量、调度、历史等功能。
